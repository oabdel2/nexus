Documentation

Everything you need to integrate, configure, and deploy Nexus — the intelligent AI inference gateway that routes requests to the optimal model based on complexity, cost, and context.

Quick Start

Get Nexus running in under two minutes. The only prerequisite is Docker.

bash
# 1. Pull and run
docker compose up -d

# 2. Send your first request
curl http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "X-Workflow-ID: my-first-workflow" \
  -H "X-Step-Number: 1" \
  -d '{
    "messages": [{"role": "user", "content": "Hello, world!"}]
  }'

# 3. Check the dashboard
open http://localhost:8080/dashboard

That's it. Nexus will analyze the request complexity, select the most cost-effective model, and return an OpenAI-compatible response. Head to the Installation section for additional deployment options.

Installation

Nexus ships as a single binary and an official Docker image. Choose the method that fits your infrastructure.

Docker (Recommended)

Docker is the simplest way to run Nexus. The official image is published to the GitHub Container Registry and includes all required dependencies.

yaml
# docker-compose.yaml
version: '3.8'
services:
  nexus:
    image: ghcr.io/nexus-ai/nexus:latest
    ports:
      - "8080:8080"
      - "9090:9090"
    volumes:
      - ./config.yaml:/etc/nexus/config.yaml
    environment:
      - NEXUS_LOG_LEVEL=info

Run with docker compose up -d and Nexus will be available at http://localhost:8080. The metrics endpoint runs on port 9090.

Go Binary

If you prefer running Nexus directly on the host, you can download the pre-built binary or build with Go.

bash
# Download latest release
curl -sSL https://get.nexus-ai.dev | sh

# Or install with Go (requires Go 1.22+)
go install github.com/nexus-ai/nexus@latest

# Start the gateway
nexus serve --config ./config.yaml

Building from Source

Clone the repository and build locally. Useful for development or when you need custom modifications.

bash
git clone https://github.com/nexus-ai/nexus.git
cd nexus
go build -o nexus ./cmd/nexus
./nexus serve --config ./config.yaml

First Request

Nexus exposes an OpenAI-compatible chat completions endpoint. Any existing code that talks to the OpenAI API can point to Nexus instead — just change the base URL.

Nexus uses two custom HTTP headers to enable intelligent workflow-aware routing:

  • X-Workflow-ID — A string identifying the logical workflow or application. Nexus uses this to track cost per workflow and learn from feedback across requests within the same workflow.
  • X-Step-Number — An integer indicating the step within the workflow. Earlier steps in a pipeline (e.g., planning) can be routed to premium models while later, more mechanical steps (e.g., formatting) use economy models.
💡 Note

Both headers are optional. If omitted, Nexus uses the default workflow and step 1. However, providing them enables smarter routing decisions and more granular cost tracking.

Here's a complete example with a detailed response walkthrough:

bash
curl http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "X-Workflow-ID: onboarding-bot" \
  -H "X-Step-Number: 2" \
  -H "X-Team-ID: engineering" \
  -d '{
    "messages": [
      {"role": "system", "content": "You are a helpful onboarding assistant."},
      {"role": "user", "content": "Explain our CI/CD pipeline and how to set up a dev environment."}
    ],
    "temperature": 0.7
  }'

The response follows the standard OpenAI format, with additional Nexus metadata in the x-nexus-* response headers:

json
{
  "id": "chatcmpl-nexus-abc123",
  "object": "chat.completion",
  "model": "claude-3-5-haiku",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "Here's a guide to our CI/CD pipeline..."
      },
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 42,
    "completion_tokens": 387,
    "total_tokens": 429
  }
}

Nexus also returns these response headers for observability:

  • X-Nexus-Tier — The tier selected (e.g., mid)
  • X-Nexus-Model — The actual model used (e.g., claude-3-5-haiku)
  • X-Nexus-Complexity — The computed complexity score (e.g., 0.48)
  • X-Nexus-Cost — Estimated cost of this request in USD (e.g., 0.00032)

YAML Configuration Reference

Nexus is configured through a single config.yaml file. Below is a complete, annotated reference covering every available option.

yaml
# ── Server ──────────────────────────────────────
server:
  host: 0.0.0.0
  port: 8080
  metrics_port: 9090

# ── Routing ─────────────────────────────────────

  default_tier: mid

    length: 0.15
    role: 0.20
    structure: 0.15
    budget: 0.20

# ── Tiers ───────────────────────────────────────
tiers:
  economy:
    range: [0.0, 0.21]

      - provider: ollama

  cheap:
    range: [0.21, 0.35]
    models:
      - provider: openai
        model: gpt-4o-mini

    range: [0.35, 0.56]

      - provider: anthropic
        model: claude-3-5-haiku
  premium:
    range: [0.56, 1.0]
    models:
      - provider: openai
        model: gpt-4o
      - provider: anthropic
        model: claude-sonnet-4

# ── Providers ───────────────────────────────────
providers:
  openai:
    api_key: 
    base_url: https://api.openai.com/v1
  anthropic:
    api_key: 
    base_url: https://api.anthropic.com/v1
  ollama:
    base_url: http://localhost:11434

# ── Budgets ─────────────────────────────────────
budgets:

# ── Logging ─────────────────────────────────────

  level: info

Provider Setup

Nexus supports multiple LLM providers out of the box. Each provider requires minimal configuration — typically just an API key and optional base URL override.

OpenAI

The most common provider. Supports all OpenAI chat models including GPT-4o and GPT-4o-mini.

yaml
providers:
  openai:
    api_key: 
    base_url: https://api.openai.com/v1   # default, can be omitted
    timeout: 60                              # request timeout in seconds
    max_retries: 3                          # retry on 5xx errors

Anthropic

Native Anthropic API support. Nexus automatically translates between the OpenAI request format and the Anthropic Messages API.

yaml
providers:
  anthropic:
    api_key: 
    base_url: https://api.anthropic.com/v1  # default, can be omitted
    api_version: 2024-01-01                 # Anthropic API version header

Ollama (Local Models)

Use locally-hosted models through Ollama for the economy tier. No API key required — just ensure Ollama is running and accessible.

yaml
providers:
  ollama:
    base_url: http://localhost:11434
    # No API key needed for local Ollama

# Make sure the model is pulled first:
# ollama pull llama3.2:3b

POST /v1/chat/completions

The primary endpoint. Accepts requests in the OpenAI Chat Completions format and returns OpenAI-compatible responses. Nexus scores the request complexity, checks budget constraints, selects the optimal tier, and forwards to the appropriate provider.

Request Headers

HeaderRequiredDescription
Content-TypeYesMust be application/json.
X-Workflow-IDNoLogical workflow identifier for cost tracking and routing optimization.
X-Step-NumberNoInteger step within the workflow. Influences routing decisions.
X-Team-IDNoTeam identifier for per-team budget enforcement.
X-Budget-IDNoCustom budget identifier. Overrides team-based budget lookup.

Request Body

json
{
  "messages": [
    { "role": "system", "content": "You are a helpful assistant." },
    { "role": "user", "content": "Summarize this article." }
  ],
  "temperature": 0.7,
  "max_tokens": 1024,
  "stream": false
}

All standard OpenAI parameters are supported: temperature, top_p, max_tokens, stream, stop, presence_penalty, frequency_penalty, and n. They are forwarded to the selected provider as-is.

Response

Returns a standard OpenAI Chat Completion object. The model field reflects the actual model that served the request, not necessarily what was requested.

POST /v1/feedback

Submit quality feedback after receiving a response. Nexus uses this data to improve routing decisions over time — if a particular type of request consistently gets low ratings on a cheaper model, Nexus will learn to route similar requests to a higher tier.

Request Body

json
{
  "workflow_id": "onboarding-bot",
  "step_number": 2,
  "rating": 4,
  "comment": "Good response but slightly verbose."
}
FieldTypeDescription
workflow_idstringThe workflow that generated the response.
step_numberintegerThe step within the workflow.
ratingintegerQuality rating from 1 (poor) to 5 (excellent).
commentstringOptional free-text feedback.

Returns 204 No Content on success.

GET /health

A lightweight health check endpoint for load balancers and orchestrators. Returns the current status, version, and uptime.

json
{
  "status": "healthy",
  "version": "0.1.0",
  "uptime": "24h3m"
}

Returns HTTP 200 when healthy, 503 when the gateway is degraded (e.g., all providers unreachable). Use this for Kubernetes liveness probes.

GET /metrics

Exposes Prometheus-compatible metrics on the configured metrics port (default 9090). Scrape this endpoint with Prometheus, Datadog, or any compatible collector.

Key Metrics

MetricTypeDescription
nexus_requests_totalCounterTotal number of requests processed, labeled by tier, provider, model, and status.
nexus_routing_duration_secondsHistogramTime spent on complexity scoring and tier selection (excludes provider latency).
nexus_provider_duration_secondsHistogramEnd-to-end latency for each provider request, labeled by provider and model.
nexus_cost_totalCounterCumulative estimated cost in USD, labeled by team, workflow, and tier.
nexus_tier_usageCounterNumber of requests per tier — useful for understanding tier distribution.
nexus_budget_remainingGaugeRemaining budget for each team/workflow as a percentage.
nexus_complexity_scoreHistogramDistribution of complexity scores across requests.

GET /dashboard

Nexus includes a built-in web dashboard at /dashboard that provides a real-time view of your gateway's operation. The dashboard is served from the main API port — no additional configuration required.

The dashboard displays:

  • Live Request Feed — Real-time stream of incoming requests with complexity scores, tier selections, and latency.
  • Cost Tracking — Daily and monthly cost breakdown by team, workflow, and tier with trend charts.
  • Budget Status — Visual indicators for each team's budget consumption with alerts for approaching limits.
  • Tier Distribution — Pie and bar charts showing how requests are distributed across model tiers.
  • Provider Health — Latency percentiles and error rates for each configured provider.
💡 Note

The dashboard is read-only and requires no authentication by default. In production, place Nexus behind a reverse proxy and restrict access to /dashboard as needed.

Python SDK Installation

The official Python SDK provides a high-level client with workflow context management, automatic retries, and integrations for popular frameworks.

bash
# Install from PyPI
pip install nexus-gateway

# With optional integrations
pip install nexus-gateway[langchain,crewai]

Requires Python 3.9 or later. The SDK has minimal dependencies — only httpx and pydantic.

NexusClient

The core client class. Manages connections, retries, and workflow state.

python
from nexus import NexusClient

client = NexusClient(
    base_url="http://localhost:8080",
    default_workflow="my-app"
)

# Simple request
response = client.chat(
    messages=[{"role": "user", "content": "What is 2+2?"}],
    step=1
)
print(response.content)       # "4"
print(response.model)         # "llama3.2:3b" (economy tier — simple question)
print(response.tier)          # "economy"
print(response.cost)          # 0.0 (local model)

# With workflow context manager
with client.workflow("data-pipeline") as wf:
    plan = wf.chat("Plan the ETL pipeline", step=1)
    code = wf.chat("Write the extraction code", step=2)
    test = wf.chat("Write unit tests", step=3)

# Async support
import asyncio
from nexus import AsyncNexusClient

async def main():
    client = AsyncNexusClient(base_url="http://localhost:8080")
    response = await client.chat(
        messages=[{"role": "user", "content": "Hello!"}]
    )
    print(response.content)

asyncio.run(main())

LangChain Integration

Drop-in replacement for LangChain's LLM classes. Nexus handles routing transparently while LangChain manages the chain logic.

python
from nexus.integrations import NexusLangChain
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

# Initialize Nexus as a LangChain LLM
llm = NexusLangChain(
    base_url="http://localhost:8080",
    workflow_id="langchain-app"
)

# Use in any LangChain chain
prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a concise assistant."),
    ("user", "{input}")
])
parser = StrOutputParser()

chain = prompt | llm | parser
result = chain.invoke({"input": "What is machine learning?"})

CrewAI Integration

Use Nexus as the LLM backend for CrewAI agents. Nexus automatically tracks each agent's step within the crew workflow and routes accordingly — planning agents get premium models, execution agents get economy models.

python
from nexus.integrations import NexusCrewAI
from crewai import Agent, Task, Crew

# Initialize Nexus as the LLM
llm = NexusCrewAI(base_url="http://localhost:8080")

# Define agents — Nexus routes each one optimally
researcher = Agent(
    role="Senior Researcher",
    goal="Find and analyze relevant data",
    backstory="Expert at research and analysis.",
    llm=llm  # Nexus automatically tracks crew steps
)

writer = Agent(
    role="Technical Writer",
    goal="Write clear documentation",
    backstory="Skilled technical communicator.",
    llm=llm
)

# Create tasks and crew
research_task = Task(
    description="Research the latest trends in AI inference.",
    agent=researcher,
    expected_output="A summary of current trends."
)
write_task = Task(
    description="Write a blog post based on the research.",
    agent=writer,
    expected_output="A polished blog post."
)

crew = Crew(agents=[researcher, writer], tasks=[research_task, write_task])
result = crew.kickoff()

How Routing Works

Every request to Nexus flows through a deterministic pipeline. Understanding this pipeline helps you configure tiers and budgets for optimal results.

📥 Request
→
📋 Header Extraction
→
🧮 Complexity Scoring
→
💰 Budget Check
→
🎯 Tier Selection
→
🔀 Provider Routing
→
📤 Response
  1. Header Extraction — Nexus reads X-Workflow-ID, X-Step-Number, X-Team-ID, and X-Budget-ID from the request. These provide context for routing and budget decisions.
  2. Complexity Scoring — The request body is analyzed using the CASTER-inspired algorithm (see below). The result is a score between 0.0 and 1.0.
  3. Budget Check — If the team or workflow is approaching its budget limit, the score is adjusted downward to favor cheaper tiers. At 100% budget consumption, requests are automatically downgraded to the economy tier.
  4. Tier Selection — The final score is mapped to a tier using the configured ranges (e.g., 0.0–0.21 = economy, 0.21–0.35 = cheap, etc.).
  5. Provider Routing — Within the selected tier, Nexus picks a provider based on availability and latency. If multiple models are configured in a tier, Nexus load-balances across them.
  6. Response — The provider's response is returned to the client with additional X-Nexus-* headers for observability.

Complexity Scoring

The complexity scoring engine is inspired by the CASTER (Complexity-Aware Smart Tier-based Efficient Routing) research. It analyzes five dimensions of each request and produces a composite score between 0.0 (trivial) and 1.0 (highly complex).

Scoring Dimensions

1. Keyword Analysis (weight: 0.30)

The message content is scanned for keywords that indicate complexity level:

  • High-complexity keywords — "analyze", "compare", "evaluate", "architect", "design", "optimize", "debug", "refactor", "explain in detail", "trade-offs"
  • Medium-complexity keywords — "summarize", "describe", "implement", "write", "create", "list"
  • Low-complexity keywords — "hello", "thanks", "yes", "no", "what is", "define"

The keyword score is calculated as the ratio of high-complexity keywords to total keywords found, normalized to the [0, 1] range.

2. Length Scoring (weight: 0.15)

Longer messages tend to contain more nuanced requests. The length score is based on total token count across all messages, with a logarithmic curve that plateaus around 2000 tokens.

3. Role Weight (weight: 0.20)

System messages with detailed instructions signal more complex tasks. This dimension considers the number of messages, the presence and length of system prompts, and the depth of the conversation.

4. Structure Analysis (weight: 0.15)

Requests containing code blocks, JSON, bullet lists, or multi-part questions receive higher structure scores. This captures the structural complexity that keyword analysis alone might miss.

5. Budget Pressure (weight: 0.20)

As a team or workflow approaches its budget limit, this dimension increases, pushing the overall score downward and favoring cheaper tiers. At 80% budget consumption (configurable via downgrade_at), the pressure starts affecting routing.

Final Formula

bash
score = keyword × 0.30 + length × 0.15 + role × 0.20 + structure × 0.15 + budget × 0.20

# Example: "Explain the trade-offs between microservices and monoliths"
# keyword  = 0.85 (high-complexity: "explain", "trade-offs")
# length   = 0.15 (short message)
# role     = 0.10 (single user message, no system prompt)
# structure = 0.05 (plain text, no code or lists)
# budget   = 0.50 (team at 40% of monthly budget)

# score = 0.85×0.30 + 0.15×0.15 + 0.10×0.20 + 0.05×0.15 + 0.50×0.20
# score = 0.255 + 0.0225 + 0.02 + 0.0075 + 0.10 = 0.405
# → Tier: "mid" (range 0.35–0.56) → Model: claude-3-5-haiku
💡 Tip

You can customize the weights in your config.yaml under routing.complexity_weights. The weights must sum to 1.0.

Budget System

Nexus provides fine-grained cost controls through its budget system. Budgets can be set at the organization level (default), per team, or per workflow.

Budget Hierarchy

  • Default Budget — Applied to all requests that don't match a team or workflow-specific budget. Set under budgets.default.
  • Team Budgets — Override the default for requests with a matching X-Team-ID header. Configured under budgets.teams.
  • Workflow Budgets — Most granular level. Override both default and team budgets for requests with a matching X-Budget-ID header.

Automatic Downgrade Behavior

When a budget reaches the downgrade_at threshold (default: 80%), Nexus begins progressively routing requests to cheaper tiers. The budget pressure component of the complexity score increases linearly from the threshold to the limit:

  • 0–80% consumed — No downgrade pressure. Routing is based purely on request complexity.
  • 80–100% consumed — Increasing downgrade pressure. Complex requests may be routed to mid instead of premium.
  • 100% consumed — All requests are routed to the economy tier regardless of complexity. A X-Nexus-Budget-Exceeded: true header is included in responses.

Budget Reset Schedules

Daily budgets reset at midnight UTC. Monthly budgets reset on the 1st of each month at midnight UTC. The reset schedule is not configurable in the current version — future releases will support custom reset cron expressions.

yaml
# Example: Strict budget for the support team
budgets:
  default:
    max_daily: 100.00
    max_monthly: 2000.00
    downgrade_at: 80%       # start downgrading at 80% usage
  teams:
    engineering:
      max_monthly: 5000.00   # higher limit for engineering
      downgrade_at: 90%      # more headroom before downgrade
    support:
      max_monthly: 500.00    # conservative limit for support
      max_daily: 25.00       # with daily cap to prevent spikes
